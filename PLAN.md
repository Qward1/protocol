## Общий контекст (входит в каждый промпт)

Проект: «Цифровой Офис» — платформа работы со встречами на русском языке:
загрузка аудио/видео → ASR (OpenRouter, `google/gemini-2.5-flash`) → протокол +
поручения (Dify workflow по HTTP) → реестр поручений с контролем исполнения →
Q&A и семантический поиск. Бот MAX — опционален (`max.enabled`).

Стек: `backend/` — FastAPI + SQLAlchemy + SQLite (без Alembic, без очередей),
`frontend/` — React 18 + Vite + TS + Tailwind (CSS-переменные тем в
`src/index.css`) + react-query + zustand + react-router v6.

**Жёсткие ограничения (из CLAUDE.md, нарушать нельзя):**
1. LLM-логика по тексту — только в Dify workflow; ffmpeg/хранение/оркестрация — только в backend. Не переносить между ними.
2. Конфигурация — только YAML (`config.yaml`), никаких `.env`/`python-dotenv`.
3. Нет Alembic: новые колонки — через `Base.metadata.create_all` + ручной идемпотентный список `ALTER TABLE` в `backend/app/db.py` (`_TASK_ADDED_COLUMNS`-паттерн).
4. Нет очереди задач: ASR на `BackgroundTasks` — известное ограничение, не «чинить» заменой на Celery/RQ без обсуждения.
5. Отсутствующие внешние ключи (OpenRouter/Dify/MAX) не должны приводить к 500 — «мягкая» деградация (пустой результат + warning в лог).
6. `frontend/src/lib/api.ts` — ручное зеркало `backend/app/schemas.py`: при изменении DTO менять оба.
7. Все URL фронтенда — корневые (`/api/...`), `baseURL = import.meta.env.BASE_URL`; не хардкодить `http://localhost:8080`.
8. При добавлении эндпоинта — обязательный `Depends(require_permission(...))`; фронтовое `can()` — только UI-гейт.
9. Проверка после изменений: `cd backend && python -m pytest -q` и `cd frontend && npm run build` (это и есть «тест» фронтенда).

### 1.1 [КРИТИЧНО, безопасность] Path traversal в SPA-фоллбэке
- **Где:** `backend/app/main.py:169-178` — `spa(full_path)` делает `candidate = _DIST / full_path` и отдаёт файл без проверки, что путь остался внутри `dist`.
- **Почему проблема:** процентно-закодированный путь (`/%2e%2e/…`) декодируется Starlette в параметр `full_path` вида `../backend/config.yaml` — файл вне `dist` (включая `config.yaml` с ключами) может быть отдан наружу.
- **Как исправить:** резолвить кандидата и проверять принадлежность каталогу:
  `resolved = candidate.resolve(); if not resolved.is_relative_to(_DIST.resolve()): return index.html` (или 404). Добавить тест с `client.get("/%2e%2e/config.yaml")` и с обычным легитимным ассетом.
- Заодно: неизвестные пути `/api/*` сейчас проваливаются в catch-all и возвращают **HTML со статусом 200**. Для `full_path.startswith("api/")` возвращать 404 JSON.

### 1.2 [КРИТИЧНО] Сломанный тест `test_send_task_to_max_disabled`
- **Где:** `backend/tests/test_api.py:83` ждёт `"MAX отключён"`, а `backend/app/api/tasks.py:142` возвращает `"MAX не настроен: max.enabled=false"`. Подтверждено запуском pytest — тест падает.
- **Как исправить:** синхронизировать текст (правильнее поправить ассерт теста на `"MAX не настроен"` — сообщение эндпоинта информативнее). Ничего не менять в логике.

### 1.3 [КРИТИЧНО] Тесты anyio параметризуются на trio, которого нет
- **Где:** `backend/tests/test_units.py:107,119` (`@pytest.mark.anyio`) — anyio по умолчанию гоняет и `asyncio`, и `trio`; trio не в `requirements.txt` → 2 гарантированных падения в чистом окружении (подтверждено).

### 1.4 [КРИТИЧНО] Смешение таймзон в отметках времени
- **Где:**
  - `backend/app/models.py:_now()` — наивный **UTC** (default для `created_at`);
  - `backend/app/api/tasks.py:116` — `closed_at = _now()` (**UTC**);
  - `backend/app/services/max_handler.py:165,262` и `reminders.py` — `now_local_naive()` (**Europe/Moscow**).
- **Почему проблема:** одно и то же поле (`closed_at`, `notified_at`) получает время в разных зонах в зависимости от пути закрытия задачи (веб vs MAX) — расхождение на 3 часа в данных и в UI.
- **Как исправить:** выбрать одну политику — хранить наивный UTC везде (минимальный дифф: в `max_handler.py`/`reminders.py` для **полей БД** использовать `_now()` из models; `now_local_naive()` оставить только для сравнения с дедлайнами, которые вводятся в местном времени). Задокументировать выбор комментарием у `_now()`.

### 1.5 [КРИТИЧНО] Даты в UI показываются со сдвигом на таймзону
- **Где:** backend отдаёт наивные UTC-datetime без `Z` (`2026-07-19T09:00:00`); `frontend/src/lib/utils.ts:fmtDate` делает `new Date(iso)` — строка без смещения трактуется JS как **локальное** время. Итог: все `created_at`/`closed_at` показываются на «минус смещение зоны» (для МСК — на 3 часа раньше реального).
- **Как исправить (минимально, на фронте):** в `fmtDate` дописывать `Z`, если строка соответствует ISO без зоны: `new Date(/([+-]\d\d:?\d\d|Z)$/.test(iso) ? iso : iso + "Z")`. Альтернатива на бэке — сериализовать с `Z` (aware UTC) — но тогда проверить все места сравнения naive datetime.

### 1.6 [ВЫСОКО] Админ может снять с себя роль последнего администратора
- **Где:** `backend/app/api/auth.py:update_user` — защита есть только в `delete_user` (последний админ, самоудаление). PATCH позволяет `admin` сменить себе роль или поставить `is_active=false`, оставив систему без администраторов.
- **Как исправить:** в `update_user` запретить (400) понижение роли/деактивацию, если целевой пользователь — последний активный админ. Тест.

### 1.7 [ВЫСОКО] Ошибка генерации протокола «молча» создаёт пустой протокол
- **Где:** `backend/app/api/protocols.py:52-83` — `Protocol(title="Генерация…")` коммитится ДО вызова Dify; при недоступном Dify `run_command` мягко возвращает пусто → протокол сохраняется с заголовком-фоллбэком «Протокол совещания», без body и задач; фронт (`TranscriptPage.genProtocol.onSuccess`) уводит пользователя на пустой протокол без объяснений.
- **Почему проблема:** пользователь не понимает, что произошло; БД засоряется пустышками; это нарушает собственный принцип «ошибка на прямом действии пользователя должна быть видна».
- **Как исправить:** если `result.raw.get("error")` и `not result.answer` — удалить созданный протокол и вернуть 502 с человекочитаемым detail («Dify недоступен / не настроен — проверьте /api/health»). На фронте показать ошибку мутации (сейчас `onError` не обрабатывается). Тест: mock `dify_client.run_command` → error → 502 и отсутствие протокола в БД.

# ПРОМПТ 2 — UX-сценарии и подсистема сроков («календарь»)

### 2.1 Единый источник истины для сроков (фундамент «календаря»)
**Текущее состояние:** `Task.deadline` — свободная строка от LLM («до пятницы», «25.12.2025»). Парсинг дат продублирован и расходится: `backend/app/services/reminders.py:parse_deadline` (8 форматов) и `frontend/src/lib/utils.ts:DEADLINE_FORMATS` (2 формата). Сортировки по сроку нет нигде; напоминания молча пропускают нераспознанные строки; фронт часть валидных дат не распознаёт (например `2025-12-25 09:00` распознаёт, а «25 декабря» — никто).

**Ограничения текущей реализации:** нельзя сортировать/группировать по сроку; «просрочено» вычисляется на глаз двумя разными алгоритмами; исполнитель не видит, распознан ли его срок системой напоминаний.

**Что сделать:**
1. Добавить колонку `deadline_at: DateTime | None` в `Task` (+ запись в `_TASK_ADDED_COLUMNS`).
2. При создании задач из протокола (`protocols.py:_apply_dify_protocol`) и при PATCH (`tasks.py:update_task`, если менялся `deadline`) вычислять `deadline_at = parse_deadline(deadline)` — реюз существующего парсера reminders (перенести `parse_deadline` в отдельный модуль `app/services/deadlines.py`, чтобы не тянуть MAX-зависимости).
3. Отдавать `deadline_at` в `TaskDTO` (+ зеркало в api.ts). Фронтовые `deadlineUrgency`/`deadlineColor` переключить на `deadline_at`, если он есть; строковый парсинг оставить фоллбэком.
4. `reminders.scan_once` читать `deadline_at` вместо повторного парсинга.

**Зачем:** одна логика вместо двух расходящихся; появляется сортировка/группировка; напоминания и подсветка перестают спорить друг с другом.

### 2.2 Редактирование срока — date-picker вместо голого текста
**Где:** `DashboardPage.tsx:EditForm` — поле «Срок» это `<input className="input">` со свободным текстом.
**Что сделать:** составное поле: нативный `<input type="date">` (+опционально `type="time"`) рядом с текстовым представлением; при выборе даты писать в `deadline` каноничный формат `ДД.ММ.ГГГГ [ЧЧ:ММ]`. Свободный ввод сохранить (LLM-строки бывают осмысленными), но при распознавании показывать бейдж «✓ 25.12.2025 18:00», при нераспознанном — предупреждение «срок не распознан — напоминание не сработает».
**Зачем:** убирает главный источник ошибок ввода; делает поведение напоминаний предсказуемым и видимым.

### 2.3 Реестр поручений: сортировка, фильтр «Просрочено», счётчик
**Текущее состояние:** задачи всегда отсортированы по `created_at desc`; фильтры — только по статусу; просроченные никак не выделены на уровне списка (только цвет текста срока в карточке).
**Что сделать:**
- Сортировка-переключатель: «по дате создания» (default) / «по сроку» (NULL-ы в конец).
- В стат-карточки добавить «Просрочено» (клик — фильтр), убрать некликабельность «В MAX» (сделать фильтром «отправленные в MAX») — сейчас 4 из 5 карточек кликабельны, одна нет, это дезориентирует.
- Фильтр статуса и поисковый запрос отражать в URL (`useSearchParams`) — сейчас состояние теряется при переходе к протоколу и «назад».
**Зачем:** реестр — главный рабочий экран; сейчас найти «горящее» можно только глазами.

### 2.4 Достроить jump-to-fragment (сейчас — мёртвый код)
**Текущее состояние:** `usePlayer` (store/selection.ts:36-47) не используется ни одним компонентом; `Citation.start` и `SearchHit.start` заполняются на бэке, но UI по клику ведёт просто на страницу записи без перемотки. CLAUDE.md декларирует «jump-to-fragment» как фичу, от которой зависит фронт — фактически она не реализована.

- Удалить `usePlayer` или использовать его же для внутристраничной перемотки — не оставлять мёртвым.
**Зачем:** это «магия» продукта — от ответа/поручения к живому месту записи за один клик; сейчас обещана, но не работает.

### 2.5 Вернуть в UI существующие backend-возможности
**Текущее состояние:** `api.retryTranscription` и `api.updateSegments` объявлены в api.ts, эндпоинты работают, но **ни одна страница их не вызывает**: после ошибки ASR пользователь видит красный текст и тупик; правка текста сегментов невозможна.
