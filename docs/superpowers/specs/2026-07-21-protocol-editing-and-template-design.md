# Дизайн: редактирование протокола + редактируемый DOCX-шаблон

Дата: 2026-07-21
Статус: согласован с пользователем, готов к реализации

## Контекст

Два новых требования к "Цифровому Офису":
1. Пользователь может отредактировать текст протокола перед сохранением/выгрузкой; правки сохраняются в БД.
2. Редактируемый DOCX-шаблон протокола: фиксированная структура (разделы/переменные),
   подстановка значений, простая замена файла шаблона с автораспознаванием полей,
   и ручная правка маппинга полей в интерфейсе.

## Требование 1 — редактирование протокола

Правки поручений (`Task`) уже покрыты существующим UI (`frontend/src/components/tasks/TaskEditForm.tsx`).
Требование 1 — это правка метаданных/текста самого `Protocol`: `title`, `date`, `number`, `body`.

- Backend: `PUT /api/protocols/{id}` + `ProtocolUpdate` DTO (все поля опциональны),
  `Depends(require_permission("protocols.manage"))` — по образцу `update_segments`
  в `backend/app/api/transcriptions.py:285`. Прямая перезапись полей + `db.commit()`.
- Frontend: на `ProtocolPage.tsx` кнопка «Редактировать» → режим правки (инпуты
  title/date/number + textarea body) → мутация → инвалидация `["protocol", id]`.
- Экспорт ничего не меняет: `exporter.protocol_to_md` уже читает `title/date/number/body`
  из `Protocol`, так что правки автоматически попадают во все форматы (`docx/pdf/md/txt/json`).
- Без истории правок/аудита — простая перезапись текущего состояния (согласовано с пользователем).

## Требование 2 — редактируемый DOCX-шаблон

**Движок**: `docxtpl` (Jinja2-в-Word) — новая зависимость (`backend/requirements.txt`).

**Данные** — новая таблица `ProtocolTemplate` в `app/models.py`:
```
id: str (pk, uuid)
name: str
docx_path: str
is_active: bool
detected_placeholders_json: str   # список найденных при загрузке плейсхолдеров
field_mapping_json: str           # {canonical_field: template_placeholder}
created_at: datetime
```
Одна активная запись одновременно (при активации новой — деактивировать предыдущую).
Старые записи не удаляются (история версий "для галочки", без отдельного UI отката,
кроме возможности заново отметить старый файл активным).

**Канонический каталог полей** (фиксированный список в `app/services/protocol_template.py`,
ограничен тем, что реально есть в `Protocol`/`Task` — ничего нового не выдумывать):
`title`, `date`, `number`, `body`, и цикл `tasks[]` с `assignment`, `responsible`,
`department`, `deadline`, `status`.

**Загрузка шаблона** — `POST /api/protocol-templates`, permission `templates.manage`
(новое право, только роль `admin`):
1. Сохранить `.docx` в `storage/templates/`.
2. `docxtpl.DocxTemplate(...).get_undeclared_template_variables()` → список плейсхолдеров
   (это и есть "автораспознавание").
3. Авто-сопоставление точным совпадением имени с каноническим каталогом; несовпавшие
   остаются `unmapped` в `field_mapping_json`.
4. Деактивировать предыдущий активный шаблон, новый — активным.

**UI управления шаблоном** — новая страница, доступна только `admin` (permission
`templates.manage`), аналог `AdminUsersPage.tsx`:
- Список найденных плейсхолдеров + выпадающий список "какому полю соответствует"
  (из канонического каталога) — правится вручную, без пересборки файла.
- Кнопка загрузки нового `.docx`, скачивание текущего активного файла.

**Рендеринг при экспорте** — `exporter.py`: новая функция
`render_protocol_docx(protocol, template)` строит контекст `{placeholder: value}` из
`Protocol`/`Task` по `field_mapping_json`, рендерит через `docxtpl`.
`POST /api/export` (`object_type=protocol, fmt=docx`) использует её, если есть активный
шаблон; иначе — мягкий fallback на текущий generic `_write_docx(md)` (никаких 500-к
при отсутствии шаблона — по духу проекта, см. CLAUDE.md "мягкие" деградации).

`protocol.docx_path`, приходящий из Dify-плагина `document-generator`, для протокола
**больше не используется** при экспорте в DOCX. Сам Dify workflow не трогаем — замена
только на стороне backend/export (рендеринг документа — не LLM-логика, значит зона
backend/Платформы, а не Dify, согласно CLAUDE.md).

## Тестирование
- Backend: `PUT /api/protocols/{id}`; загрузка шаблона + автодетект + маппинг +
  рендер докса с фикстурным `.docx` в `tests/`.
- Frontend: `npm run build` (типчек — единственный "тест" фронта), плюс ручная проверка
  в браузере (правка протокола, загрузка шаблона, правка маппинга, экспорт).

## Согласованные решения (из диалога с пользователем)
- Движок шаблона: docxtpl / Jinja-плейсхолдеры (не bookmarks/content controls).
- Новый шаблон полностью заменяет DOCX-выгрузку протокола (Dify document-generator
  для этой цели больше не используется).
- Один активный шаблон одновременно (не несколько именованных).
- Управление шаблоном — только роль `admin`.
- Правки протокола — без истории/аудита, простая перезапись.
